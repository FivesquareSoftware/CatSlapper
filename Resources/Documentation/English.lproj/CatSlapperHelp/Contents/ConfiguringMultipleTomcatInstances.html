Configuring Multiple Tomcat Instances

It is possible to have multiple instance of multiple versions of Tomcat running on a single Mac OS X machine. This is often necessary when developing Java Web applications for deployment to various servers and target JDKs. CatSlapper makes it simple by capturing the necessary information to uniquely control an instance of Tomcat in each configuration you create.

The factors that make an instance of Tomcat unique from other instances are as follows:

  • The value of the environment variable CATALINA_HOME. This indicates where the Tomcat binaries are installed.
  • If two instances share the same value for CATALINA_HOME, the the value of CATALINA_BASE can differentiate them. CATALINA_BASE indicates where resources for a running instance are stored, such as logs, libraries and deployed Web applications. When CATALINA_BASE is different from CATALINA_HOME, Tomcat binaries are loaded from CATALINA_HOME and can be shared. If CATALINA_BASE is not set it defaults to the value of CATALINA_HOME.

As long as these environmental conditions are met, an instance of Tomcat can be uniquely controlled. This is true whether you are using CatSlapper to manage your Tomcats or not. CatSlapper can make it easier to correctly configure your Tomcats by warning you of conflicts or misconfigurations that may cause unexpected results when stopping, starting or performing other management tasks on a Tomcat instance.

Click the grey information button that appears next to the read text "Configuration Errors" to see why a Tomcat instance does not validate.

While not strictly part of a unique Tomcat instance configuration, setting up which ports Tomcat listens on for requests over various protocols (such as HTTP or AJP), is crucially important. This is because only one Tomcat can listen on a given port at any one time. Currently, CatSlapper does not manage these port settings for you, except when you use it to install a Tomcat instance in the first place.
